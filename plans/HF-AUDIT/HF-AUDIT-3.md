## Problem

After HF-CLEAN landed (hyperforge `4dacf7c`, v4.1.2) and the running server was restarted with the fresh release binary (PID 18804 on port 44104), synapse navigation into any child activation under hyperforge fails:

```
$ synapse --port 44104 --no-cache lforge hyperforge build
Fetch error at hyperforge.build: Protocol error: No schema in response
```

Schema fetch for the root activation (`hyperforge`) succeeds. Schema fetch for the child `build` fails.

Reproduced for `hyperforge.build`; likely reproduces for `hyperforge.workspace`, `hyperforge.repo` as well (all three use the `#[plexus_macros::child]` accessor pattern introduced in HF-CLEAN).

Raw RPC probes:

```
$ echo '{"jsonrpc":"2.0","id":1,"method":"hyperforge.build.schema","params":{}}' | websocat -t ws://127.0.0.1:44104/
{"jsonrpc":"2.0","id":1,"error":{"code":-32601,"message":"Method not found"}}
```

So the wire-level method `hyperforge.build.schema` is not registered on the server. Synapse expects that method to exist when it navigates to a child activation â€” it needs the child's `PluginSchema` to render the tree.

## Likely root cause

`#[plexus_macros::activation]` auto-emits a `schema` method on the activation impl (returns `PluginSchema`). For HF-CLEAN's `HyperforgeHub`, `schema` is at `hyperforge.schema`. For the child `BuildHub`, the synthesized method should be at `BuildHub.schema()` â€” but routed via `hyperforge.build.schema` through `HyperforgeHub`'s `ChildRouter` impl.

Hypotheses (each worth verifying):

1. **ChildRouter doesn't forward `schema` method calls.** The router generated by `#[plexus_macros::child]` accessors may only route *user-tagged* `#[method]`s to the child, not the auto-synthesized `schema()` / `_info` / `hashes` / `_protocol` introspection methods.
2. **BuildHub doesn't have `#[plexus_macros::activation]`** (maybe it's a plain struct with `#[plexus_macros::method]` blocks but no activation-level macro). If so, there's no `schema` method synthesized on it at all.
3. **ChildRouter routes the call but `BuildHub.schema()` returns `None`/empty** because `BuildHub` carries no state of its own (per the doc comment at `hubs/build/mod.rs` line near `fn build`).

## Proposed fix

Once root cause is identified, either:

- **plexus-macros fix**: `#[plexus_macros::child]` ChildRouter codegen forwards `schema`, `_info`, `hashes`, `_protocol` to the child activation (not just user methods). New plexus-macros patch version.
- **hyperforge fix**: each child hub (BuildHub, WorkspaceHub, RepoHub, etc.) gets an explicit `#[plexus_macros::activation]` or equivalent that guarantees the synthesized schema method exists and the parent's ChildRouter forwards to it.

## Resolution (2026-04-20)

Fixed by **HF-AUDIT-4** (plexus-macros 0.5.2, commit `7eb06df` â€” see `plans/HF-AUDIT/HF-AUDIT-4.md`).

Root cause was **hypothesis 1** variant: the `#[plexus_macros::activation]` macro-generated `Activation::call` `_` arm contained a `.schema` strip-suffix shortcut that returned `SchemaResult::Method` for ANY method name matching the prefix, including `#[plexus_macros::child]` accessors (which since CHILD-8 / IR-10 appear in `plugin_schema.methods` with role `StaticChild` / `DynamicChild { .. }`). That short-circuited `route_to_child`, so the caller got the parent's child-accessor MethodSchema (content_type `*.method_schema`) instead of the child's PluginSchema (content_type `*.schema`) â€” and synapse's content-type filter rejected it as "No schema in response".

Narrow fix (HF-AUDIT-4): exclude child-role entries from the shortcut's `find` predicate. Regular `#[method]`s keep hitting the shortcut; child accessors fall through to `route_to_child`, which dispatches `"<child>.schema"` into the child activation and returns `SchemaResult::Plugin(<child_plugin_schema>)` as expected.

Verified end-to-end with hyperforge 4.1.3 (rebuilt against published plexus-macros 0.5.2 from crates.io):

- `synapse --port 44104 --no-cache lforge hyperforge build` renders BuildHub's schema tree (analyze, binstall_init, brew_formula, etc.).
- `synapse --port 44104 lforge hyperforge build dirty --path ... --all_git true` streams dirty-repo events correctly.
- `synapse --port 44104 --no-cache lforge hyperforge workspace` / `repo` drill-downs also render their schema trees.

Regression test pinned in `plexus-macros/tests/hf_audit_4_child_schema_drill_down_tests.rs`.
